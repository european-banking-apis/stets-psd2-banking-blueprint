My name is Kin Lane, this is my STETs PSD2 Banking API portal template...

STETs is doing interesting work in the UK when it comes to banking APIs. They have produced a set of banking API specifications to help move forward the banking API conversation in France.

I have been learning more about their PSD2 banking API specification, and would like to get the word out about their work, and hopefully contribute to the conversation, as well as the tooling available when it comes to banking APIs in France.

This project is meant to be a forkable project that any bank in France, or anywhere else for that matter, can publish, change, and turn into their own API portal. This project is a work in progress, and I will be releasing more features soon.

Public Data APIs

These are the APIs which provie access to the public profile, products, and services for the banks. Allowing access to some of the data that is commonly scraped for use in other 3rd party applications.

This section addresses wider appetite for data about each bank, making it so the common information about banks are accessible in a standardized, machine readable way across all banks in the UK.

Reducing the need for 3rd party providers to be scraping this data from banks, and helping ensure things are more secure.

PSD2 ASPSP services for AISP, PISP and PIISP.

This API intends to provide an interface between Account Servicing Payment Service Providers (ASPSP), Third, and Party (Payment Service) Providers (TPP). TPP may act with different roles as Account Information Service Providers (AISP), Payment Initiation Service Providers (PISP), and Payment Instrument Issuer Service Providers (PIISP). The Payment Service User (PSU) is the owner of the accounts held by the ASPSP and gives accreditations to the TPP in order to access his accounts information or initiates payment from these accounts.

This is just an OpenAPI definition, published using Swagger UI--it is not a functional API or mock yet.

Support

There are three ways that you can get support when using the API--consider one of the following channels when you have a question, comment, or some feedback regarding the API and integration.

Github Twitter Email

It is recommended that you start with submitting a Github issue, then maybe tweet at us. If your questions or comments require some privacy you should use email. Make sure you look through the existing issues before submitting a new one.
